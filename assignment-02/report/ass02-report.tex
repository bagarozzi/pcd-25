\section{Analysis}

\subsection{General Requirements}
This assignment focuses on building an asynchronous library and application to compute file system statistics. It needs to traverse a directory tree and generate a report showing the total file count along with a size distribution histogram based on predefined intervals.
We also need to include an optional GUI so the user can start, stop, and monitor the scan in real time.

\subsection{Analysis of the Problem}
Because file system inspection is \textbf{I/O-bound}, disk access operations like \texttt{readdir} and \texttt{lstat} act as the main performance bottleneck.
Running these operations sequentially would stall the CPU during I/O waits, severely degrading performance. Our main challenge is maximizing throughput by parallelizing I/O requests without blocking the execution threads.

We have to build the system using three different programming paradigms: Event-Loop, Reactive Programming (Rx), and Virtual Threads. Each approach requires a specific way of handling concurrency. 

For the optional part, since the GUI needs constant updates, the computation must be interruptible and capable of returning intermediate results.

\section{Design}
In this section is discussed the design of the three requested versions of the system. Each version is designed according to its paradigm: asynchronous programming based on event-loop, reactive programming and virtual threading.
\subsection{Asynchronous Programming (Event-Loop)}\label{sec:des-async}
Being an event-loop single-threaded, a mechanism to process multiple batches has to be implemented to avoid blocking the execution thread in
the massive amount of I/O operations that will be done. 

The main idea is to maintain a queue of directories to scan and hand it to the event queue to scan. Each directory scan will produce other 
directories that will be added to the queue, and so on.

Combining this queue with asynchronous calls, the event-loop can proceed, executing other tasks, while waiting for an async call to resolve, then return to it when done.

\subsection{Virtual Threads}\label{sec:des-vthreads}
The Virtual Threads model lets us skip complex asynchronous callbacks and reactive pipelines, allowing us to write traditional, synchronous-looking code. They act as lightweight logical units of concurrency, giving us the benefits of standard threads without the overhead brought by traditional threads.
In fact vThreads are managed directly by the JVM, which maps them to platform threads by mounting and unmouting them when they are performing blocking operations, freeing up resources for other vThreads to run on. Moreover, they do not have a stack like their platform counterparts but rather a smaller stack, improving on memory efficiency.

So, while traditional platform threads are limited by the resources available, virtual threads can be created and destroyed dinamically in the count of up to millions, fitting perfectly with the short-timed tasks required by this assignment.

The design relies on a `thread-per-task' approach where we can create a new task for each directory to explore. Each task will then add new tasks to the queue to explore each new directory found. Every tasks will be picked up by a virtual thread and executed.
When one of these threads hits a blocking I/O operation (like reading a directory), it unmounts from its OS carrier thread, leaving the carrier free to run other virtual threads. This prevents the thread-pool starvation normally seen with platform threads under heavy I/O and scales exceptionally well. 

We do have to be careful about \textit{thread pinning}, though. This happens if a virtual thread blocks inside a \texttt{synchronized} block or a native method and prevents the vThread from unmounting, killing performance.

\section{Implementation}

\subsection{Event-Loop (Node.js)}
We wrote the event-loop version mentioned in Section~\ref{sec:des-async} in Node.js to take advantage of its native non-blocking I/O. The \texttt{async-await} pattern offered by JavaScript makes writing asynchronous code much easier.

The \texttt{FSStat} class is the library's entry point and handles the core logic using the \texttt{node:fs/promises} API. Through the \texttt{getFSReport} async function the user can trigger the directory scan and obtain an \texttt{Histogram} containing all the statistics, divided 
as requested through the arguments passed.

\begin{figure}[H]

The \texttt{scanDirectory} function is also asynchronous and does the scanning. It first reads the directory passed as arguments and gets the statistics of the files. As show in Figure~\ref{fig:lsstat}, this step consist in creating a series of async lambdas calling \texttt{lsstat}, this triggers the system calls for fetching the files' statistics, which are blocking calls that are \texttt{await}-ed.

\texttt{await}-ing for the calls to \texttt{lsstat} lets the event-loop pause the execution and proceed with other tasks that may be ready after being themselves blocked for an I/O call; once the OS fetches the files' statistics, the promises are resolved and the event-loop can resume execution from where it left. 

This is particularly useful in this context as, we suppose, many other recursive calls to the function are opened and performing blocking calls and many more are done and waiting to resume.

Finally, \texttt{Promise.all\,()} call serves as aggregator, waiting for the completion of the async lambdas mentioned.

In the final part the function goes over every file, updating the report according to the file's nature. At this point \texttt{scanDirectory} should be recursively called on directories found, as shown in Figure~\ref{fig:recursion}.

We could say it acts as a \textit{barrier}.

\subsection{Virtual Threads}
We implemented the Virtual Threads version in Java. The \texttt{ScanContext} class coordinates the scanning and sets up an unbounded executor dedicated to virtual threads using \texttt{Executors.newVirtualThreadPerTaskExecutor()}.

The \texttt{DirectoryTask} class implements \texttt{Runnable} and handles the recursive traversal. When scanning a directory, it lists the contents and processes files to update the shared \texttt{Histogram}. Instead of making a blocking recursive call for subdirectories, the task submits a new \texttt{DirectoryTask} to the executor. This naturally fans out the traversal across thousands of concurrent virtual threads.

To figure out when the whole tree has been processed without blocking anything, we use an \texttt{AtomicInteger} as a concurrent counter. Each task increments this counter when created and decrements it in a \texttt{finally} block at the end of its \texttt{run()} method. When the count hits zero, the last terminating thread triggers a \texttt{CountDownLatch} to signal the scan is done. Using concurrent utilities like \texttt{AtomicInteger} and \texttt{CountDownLatch} instead of \texttt{synchronized} blocks helps the virtual threads unmount correctly and prevents thread pinning.

The Java Swing GUI (in the \texttt{View} class) integrates smoothly with this setup. Starting the scan doesn't block the Swing Event Dispatch Thread (EDT). Instead, a \texttt{ScheduledExecutorService} polls the \texttt{Histogram} every 100 milliseconds to update the UI with intermediate results. If the user stops the process, the GUI calls \texttt{executor.shutdownNow()}, which immediately interrupts the virtual threads and halts the I/O traversal cleanly.
