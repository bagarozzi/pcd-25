    \part[Extension of Smart Home Alarm System]{Smart Home Alarm System \\ \LARGE{Asynchronous message passing based on actors, distributed in a cluster using Apache Pekko}}

    \section{Analysis}
    The goal is to distribute the Smart Home Alarm System implemented in the \href{https://github.com/bagarozzi/pcd-25/tree/3f8c6f27e9e44c2a17f87c5eae827dfdc428e8c7/assignment-03}{previous assignment}.

    Even thought this is a simulation and not an actual alarm system, the architecture
    must be coherent to a real-world scenario and somehow reflect the distribution of
    sensors in a large environment such as a big mansion or an industrial plant.

    The most appropriate choice to do so is to distribute the previously modeled \textbf{Zone}s
    and their relative actors across the nodes. In this scenario, the machines running the JVM nodes
    are connected in a large network (to the central alarm node) and communicate with their
    zone's sensors throught a smaller, local network.

    The \textbf{Sensor}s would be likely uncoupled from the physical devices gathering the data,
    as running a JVM on constrained hardware such as a ESP-32, or worse an ESP-8266, is not ideal.
    These sensors may talk to the Pekko cluster throught a local network and with an appropriate protocol
    such as MQTT.\@ Given this, the \texttt{SensorActor}s may be placed in the same node as
    their relative ZoneActor, and implement communications with their physical counterpart.

    The \textbf{central alarm} acts as the central manager of the alarm, acting the same as in the previous implementation.

    Finally, the \textbf{Keypad}s have to be distributed to allow control from different physical places. This
    changes the behavior of the Keypad: previously there was only one keypad so the control was centralized, now
    the central alarm has to communicate to multiple keypads and all the keypads have to agree with the same
    states and choices in case of an alarm.

    \section{Implementation}
    In Pekko clusters, nodes are simply \texttt{ActorSystem} talking to each other, so separate ActorSystems are
    created.

    To distribute and track references to other actors, the \textbf{Sharding} extension of Pekko is used.
    Throught this extension, the actors can obtain a reference to another actor without knowing its location,
    this is done through a \textit{shard region} that acts as a router.
    To enable this, actors are not directly spawned in the node where we want them but rather is declared that
    a sharded actor \textit{can} be created on the node. This mechanism can be controlled even more throught \textit{roles}.
    By specifying the role an actor has to have to be able to create a shard, if a node hasn't got that role then a
    proxy is created to route messages to the shard.

    The implementation shards the \texttt{ZoneActor}s and the \texttt{AlarmActor}s but leaves as normal nodes
    the Keypads. The \texttt{SensorActor}s remain children of the ZoneActors, given that their behavior does not change.

    The behavior of the keypads change: a pub-sub topic is created to group together all the keypad nodes so that
    the AlarmActor can talk to them at once, the same topic is used to keypad-to-keypad communication.

    For instance, when an entry is detected in one of the areas all the keypads enter the EntryAlert state
    and when the alarm is deactivated from one terminal it has to signal the other keypads to exit the state
    or else they would remain there. This is implemented with two new messages \texttt{OtherKeypadSilenced}
    and \texttt{OtherKeypadDisarmed} and used as in Figure~\ref{fig:topic}

    A note has to be made on shard creation: sharded actors are not created immediately, but are dinamically allocated
    when they are needed, and potentially deallocated when inactive (\textit{passivated}). So when the cluster is created for testing,
    a message has to be sent to the sharded actors so that they can be created and will be available immediately.

    In \texttt{AlarmUser} a simple cluster is created, including two Keypads, three zones and the AlarmNode, that acts as
    \textbf{seed node} and is created before every other node. In the same way as the previous assignment, sensors
    are found throught the cluster's Receptionist to simulate their triggering.

    \part[Distributed TTT]{ Distributed TTT\\\LARGE{Distributed version of a Tic-Tac-Toe game with Java Remote Method Invocation}}

    \section{Analysis}

The project consists of the development of a distributed Tic-Tac-Toe game based on Java RMI. Multiple players can connect to the server and request to participate in a match. The server is responsible for pairing players by assigning a newly connected player to an existing waiting match whenever possible, or creating a new match otherwise. Since several connection requests may arrive concurrently, the matchmaking mechanism must ensure consistency and prevent synchronization issues.

Once a match starts, each pair of players interacts within its own game instance. The server must therefore support multiple matches running simultaneously while keeping their states completely independent. Actions performed in one game must not affect any other ongoing match, guaranteeing proper isolation between game sessions.

A key aspect of the system is the management of turns and shared game state. Although client requests are executed concurrently through remote method invocations, the game logic requires players to act in a strictly sequential manner. At any given time, only the player whose turn it is can perform a move, while the opponent must wait. This guarantees that moves are processed atomically and prevents race conditions that could compromise the correctness of the game.

The game board represents a shared resource accessed by both players. The system must ensure that every move is correctly reflected in the board state, that no two players can occupy the same cell, and that all game information, including the current turn and match outcome, remains consistent throughout the execution. Furthermore, updates performed by one player must become immediately visible to the opponent so that both clients always observe the same state of the game.

As the application is implemented as a distributed system using Java RMI, special attention must be paid to concurrency and synchronization. Remote requests can be handled by different server-side threads, making concurrent access to shared resources unavoidable. Appropriate synchronization mechanisms are therefore required to guarantee data consistency, correct turn management, and reliable communication between clients and server.
  \section{Design}

The system adopts a client-server architecture where the server manages 
game state and match coordination, while clients handle user interaction 
through a graphical interface.

To clearly delineate component responsibilities and facilitate synchronization, 
the design employs two distinct interface types:

\textbf{Remote interfaces} define contracts for RMI method invocations that 
cross the network boundary:
\begin{itemize}
    \item \textbf{Server}: exposed for client connection and match joining
    \item \textbf{MatchManagerRemote}: defines remote operations on a match
    \item \textbf{PlayerRemote}: defines server-to-client callbacks
\end{itemize}
These methods execute in different RMI dispatcher threads on the server, 
requiring explicit synchronization to handle concurrent access.

\textbf{Local interfaces} define contracts for in-process communication 
within the same JVM:
\begin{itemize}
    \item \textbf{Player}: GUI-to-logic communication on the client
    \item \textbf{MatchManager}: internal server-side match management
\end{itemize}
This separation of concerns ensures that each component is aware of its 
execution context and the synchronization requirements it entails. Remote 
implementations must protect against concurrent RMI calls, while local 
implementations can employ simpler patterns. As a result, race conditions 
are localized to specific synchronized methods rather than scattered 
throughout the codebase.

Critical sections on the server that receive concurrent RMI requests are 
protected through synchronized methods. This serializes access to shared 
state (match list, game board, turn management) and ensures atomicity of 
operations. The strategy maintains isolation between independent matches 
while preventing race conditions within each match.

    \section{Implementation}

The game execution begins on the client side when a player initiates a join 
request through the GUI. GamePanel invokes PlayerImpl.joinGame(), which 
performs an RMI call to the server's GameManager.

GameManager receives the remote request and executes its synchronized 
joinGame() method to ensure atomic match-pairing. It searches for an 
incomplete match; if found, assigns the player to that match, otherwise 
creates a new one. The method returns a reference to MatchManagerRemote, 
which the client stores for subsequent game operations.

\begin{verbatim}
public synchronized MatchManagerRemote joinGame(PlayerRemote player) {
    Optional<MatchManager> res = matches.stream()
            .filter(MatchManager::matchNotFull)
            .findFirst();
    if(res.isPresent()){
        res.get().addPlayer(player);
        return (MatchManagerImpl) res.get();
    }
    return createGame(player);
}
\end{verbatim}

Once both players have joined, MatchManagerImpl (the remote match handler) 
is responsible for orchestrating the game. It manages player references, 
game state, and board updates. The MatchManagerImpl.addPlayer() method is 
synchronized to prevent race conditions during player assignment and state 
initialization. Similarly, the synchronized notifyMove() method serializes 
move execution and enforces strict turn alternation, ensuring that only one 
player can act at a time.

\begin{verbatim}
public synchronized void notifyMove(Pair pos, char sign) throws RemoteException {
    if (state == State.PLAYER_1_MOVE) {
        GameBoard.result res = board.makeMove(pos, sign);
        // ... handle result
        state = State.PLAYER_2_MOVE;
    } else if (state == State.PLAYER_2_MOVE) {
        // ... similar logic
        state = State.PLAYER_1_MOVE;
    }
}
\end{verbatim}

Board updates are protected through the synchronized GameBoard.makeMove() 
method, which ensures that moves are atomic and immediately visible to all 
threads.

After each move, MatchManagerImpl sends RMI callbacks to both players 
(via PlayerRemote) to notify them of the move result and whose turn it is. 
These callbacks trigger GUI updates on the client side.

On the client side, GamePanel displays the board as a 3x3 button grid. 
When a player clicks a button, the ActionListener checks if the cell is 
empty and sends the move to the server via PlayerImpl.makeMove(). Server callbacks are received by 
PlayerImpl and forwarded to the GUI thread using SwingUtilities.invokeLater() 
to maintain Swing thread safety. Additionally, PlayerImpl declares sign as 
volatile to ensure visibility between the RMI thread that updates it and 
the GUI thread that reads it.

    \part[Distributed Critical Sections throught a MOM]{Distributed Critical Sections throught a MOM \\\LARGE{High-level, message-oriented middleware for critical sections between processes in distributed system}}

    \section{Analysis}
    The problem of distributed mutual exclusion requires coordinating multiple independent processes to ensure that only one process can access a shared critical section at any given time. In a distributed environment, traditional synchronous mechanisms like RPC (Remote Procedure Call) are often inadequate due to tight coupling and the fact that a blocking nature can severely affect scalability and performance.  To overcome these limitations, this project leverages a Message-Oriented Middleware (MOM), defined as an infrastructure that provides messaging capabilities to enable flexible communication among different applications. By introducing a middle layer that decouples participants, the MOM shifts the paradigm from a synchronous interaction to an asynchronous one, where the producer does not need to block and retains control to continue processing.  Specifically, the system requires a persistent communication model, which supports the communication of processes that are not necessarily executing at the same time. This is achieved through message queues and brokers, which are the standard choice in large, distributed, and heterogeneous systems.  To implement a robust distributed lock, the system must guarantee three fundamental properties:

    \begin{itemize}
        \item Mutual Exclusion: Only one process can hold the lock simultaneously.
        \item Fairness: Processes should acquire the lock in the order they requested it (no starvation).
        \item Liveness: The system must progress. If a process crashes while holding the lock, the system must recover and not incur a distributed deadlock.
    \end{itemize}

    \section{Design}

    The design of the distributed lock is based on a Token-Passing algorithm implemented over a MOM queue. The lock is represented by a single, unique message (the `Token') circulating within the middleware.

    \subsection*{Messaging Model}
    To guarantee mutual exclusion, the system utilizes a Point-to-Point messaging model. In this model, multiple producers and consumers can connect to a Point-to-Point Queue, but only one consumer receives the message. The queue strictly follows a First-In-First-Out (FIFO) policy.

    \subsection*{Token Management in Critical Sections}
    The lock management is divided into three logical phases to satisfy the required distributed properties:

    \begin{enumerate}
        \item \textbf{Safe Initialization (Leader Election):} When multiple asynchronous workers start simultaneously, there is a risk of generating multiple tokens. To prevent this, workers attempt to declare a temporary, exclusive `initialization queue'. The middleware guarantees that only one worker will succeed in creating it. This elected worker injects the initial token into the main queue, ensuring strict mutual exclusion from the genesis of the system.

        \item \textbf{Acquisition (Fairness):} Workers express their intent to enter the critical section by subscribing to the Point-to-Point queue as consumers. Fairness is intrinsically guaranteed by the MOM's FIFO queue: workers are queued in the exact order their subscription reaches the broker. Furthermore, the middleware is instructed to send only one unacknowledged message per worker (QoS = 1), preventing fast workers from hoarding the token.

        \item \textbf{Release and Fault Tolerance (Liveness):} The system's liveness and reliability are improved by the message persistence typically implemented in MOM nodes. Instead of immediately acknowledging (ACK) the token upon receipt, the worker holds the message in an `Unacked' state during the critical section execution.
        \begin{itemize}
            \item \textit{Normal execution:} Upon leaving the critical section, the worker explicitly rejects the message and instructs the broker to requeue it, passing it to the next waiting worker.
            \item \textit{Crash handling:} If a worker crashes while in the critical section, the TCP connection to the MOM drops. The broker detects this failure and automatically returns the `Unacked' token to the front of the queue. This completely eliminates the risk of a lost token and distributed deadlocks.
        \end{itemize}
    \end{enumerate}

    \section{Implementation}
    The system is implemented in Java using the RabbitMQ AMQP client, leveraging a modular architecture that separates configuration concerns from the domain logic.

    \subsection*{Connection Management}
    The \texttt{DistributedLockManager} acts as a factory for locks. It manages the AMQP URI credentials and generates instances of \texttt{DistributedLock} for the workers. This design promotes decoupling, allowing the system to scale independently and making the connection logic reusable across different components.

    \subsection*{Distributed Lock Core Logic}
    The \texttt{DistributedLock} class implements the low-level AMQP interaction:

    \begin{itemize}
        \item \textbf{Initialization (\texttt{init()}):} The system uses \texttt{queueDeclarePassive} to check for the existence of the main lock queue. If missing, it attempts to declare an exclusive queue named \texttt{<queueName>\_init\_lock}. RabbitMQ's atomicity ensures that only the first thread succeeds; others receive an \texttt{IOException}. The `winner' thread suppresses this exception, creates the \texttt{PERSISTENT\_TEXT\_PLAIN} token, and publishes it to the main queue using \texttt{basicPublish}. % chktex 36

        \item \textbf{Acquisition (\texttt{acquire()}):} Workers prepare to consume the token. We invoke \texttt{channel.basicQos(1)} to instruct the broker to dispatch exactly one token per channel. % chktex 36
        To ensure that the worker requesting does not poll, resulting in a busy wait, a \texttt{DeliverCallback} is registered and a \texttt{Future} is created so that the thread can wait on its completion.
        Upon token arrival, the \texttt{CompletableFuture} is resolved with the message's \texttt{deliveryTag}, allowing the main thread to unblock and enter the critical section while the message remains `unacknowledged'.

        \item \textbf{Release (\texttt{release()}):} Upon exiting the critical section, the worker executes \texttt{channel.\-basicCancel()} to unsubscribe the consumer, preventing immediate re-acquisition. Finally, it invokes \texttt{channel.\-basicReject()} to reject the message. The \texttt{requeue} flag is set to \texttt{true} and instructs the MOM to put the token back to the front of the queue, making it immediately available for the next waiting consumer.% chktex 36
    \end{itemize}
