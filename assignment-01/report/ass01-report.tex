\section{Analysis}
\subsection{General requirements}
It is requested to develop a game of pool where players control a ball and fight to
score points by pushing other, smaller, balls inside of the corner holes. 

One ball must be controlled by the player and the other controlled by the computer. At the
end of the game, the player with the most points wins.

It is required to develop two solutions: one using default Java multithread programming
and one using the Java Executor Framework. It is also necessary to use high-level constructs
such as \textbf{monitors} to manage concurrency.

\subsection{Analysis of the sequential solution}\label{sec:analysis-solution}
The sequential solution, available as \texttt{sketch01}, when run with large
quantities of balls, is not efficient because at every update,
the single thread running the loop has to cycle multiple times throught every
game object to calculate collisions and update movements.
The the loop has to wait for the view to render the frame with the new update,
resulting in a performance of about less than $\sim$20 frames per second (FPS).

A solution adopting concurrent programming could take advantage of all
the available cores of the machine thus improving calculations speed.
Also, the collisions calculations and the view's rendering could be 
approached in a more efficient way.
\subsection{Analysis of the problem}\label{analysis:prob}
% as explained in the course, first we identify the tasks, then we map them to 
% active components
The critical tasks identified are:
\begin{itemize}
    \item \textbf{movement calculations}: every ball must be moved in a new position according
        to its current speed
    \item \textbf{collision calculations}: every ball colliding with other balls must change
        direction and speed accordingly
    \item \textbf{frame rendering}: the updated game state is passed to the view that renders it.
        It's important to wait for the complete rendering of the frame before the next update.
\end{itemize}

\section{Design}\label{sec:design}
\subsection{Architecture}
The architecture adopted is the \textbf{Model-View-Controller}. The controller is composed
of all the threads needed for the computation in a \textbf{master-worker} architecture
where the control loop, a thread itself, acts as master towards the other threads
by synchronisiting their actions and giving them tasks to execute.

\subsection{Components}\label{sec:comp}
The \textbf{active components} needed are threads that will execute work on behalf 
of the controller. These components include also the thread rendering the view, in
the Java implementation known as the Event Dispatcher Thread.

\begin{figure}[H]
    \input{./figures/petri/workers-petri.tikz}
    \centering
    \caption{Petri diagram of the workers receiving and computing tasks}\label{fig:workers-petri}
\end{figure}

The \textbf{passive components} needed are:
\begin{itemize}
    \item data structures that have to be shared between the threads,
        implemented as monitors
    \item barriers, to synchronize the threads' (master and workers) activities during 
        the calculation phases 
\end{itemize}

Figure~\ref{fig:workers-petri} explains how three workers (they could be more) receive
tasks from a shared buffer and complete them; the transition '\textit{task done}' serves
as barrier so that the work is completed at once.

\subsection{Modeling}
The components mentioned in Section~\ref{sec:comp} and Section~\ref{analysis:prob} are modeled as follows:
\begin{itemize}
    \item The tasks identified in the Section~\ref{analysis:prob} are modeled as runnable functions
        so that the master can send them to the worker who will execute them.

    \item The barriers, one for each computational task, are modeled with objects exposing a
    \texttt{hit()} method so that the active components can hit it once their work is done. % chktex 36
    
    \item A barrier between the View and the Controller, as modeled in Figure~\ref{fig:vc-petri},
    is also necessary so that an update doesn't start before the rendering of the current frame.

    \item Each ball is modeled as a monitor to prevent races during the collisions calculation.

    \item The class that contains all the balls, the Board, is modeled itself as monitor to ensure
    that there's not critical races when the workers are operating on it.
\end{itemize}
Other domain entities that do not participate on concurrent activities
such as speed, position and boundary are implemented according to their
specifications.

\end{figure}


\section{Implementation}
For both versions it has been decided to create as many threads as the available 
cores to exploit all the system's resources and keep the CPU busy.
In fact creating a larger number of threads would mean more context switching
and it would prove damaging to performance.

The \textbf{tasks} are implemented as \texttt{Java.Runnable} classes and passed to
the workers throught a bounded buffer, that itself implements monitor
behaviour for concurrent access.

A \textbf{double buffer} is implemented in the \texttt{ViewFrame} class.
This mechanism uses two images: while one image is displayed, 
the other updates with the new frame and once the process is done it gets
swapped with the old frame. 
Although this does not improve directly performance, as the rendering time remains the same,
it reduces visual artifacts such as tearing because the buffer swap is instantaneous.

The barrier between the View and the Controller, referenced in Figure~\ref{fig:vc-petri}
is not implemented as a standard barrier, it instead makes calls synchronous by first
calling \texttt{wait()} and then \texttt{notify()} when the render is done. % chktex 36 

As mentioned in Section~\ref{sec:analysis-solution}, collision detection can be 
approached differently. It's introduced a \textbf{Spatial Hash Grid}: this structure creates
a virtual grid, the size of the balls' diameter, in this way every ball can be mapped
to a specific cell and its hash code will reflect its position in the grid.
This makes easier the the collision detection as the worker has to look only at 
the adjacient cells for each ball.

\section{Performance evaluation}

\subsection{Methodology}
Performance test are conducted on the final, headless, application set up with \num{4500} balls
on the board and a varying number of cores assigned for the workers: from 1 to 16.
To create a repeatable testing environment, the game loop is modified to run a finite
number of \num{10000} ticks. The high number of ticks is needed so that each test run can be distinguished from another and errors up to 
\qty{100}{\milli\second} will not interfere with the calculations.

The first \num{5000} ticks are to warm up the JVM, during this time the JIT (Just-In-Time) compiler
creates optimized versions of the \textit{hot} methods. This can also be observed in the graphic version
of the game where the FPS will rise significantly at the beginning of the execution.

The second \num{5000} ticks are measured using \texttt{System.nanoTime} and use a specific, heavy and fast
ball to scatter the other balls in a more predictable way, although it has been observed that the same collision
can produce slightly different trajectories.

Each run is performed by running the \texttt{.jar} executable file created by setting
the \texttt{TestController} class each time with the number of workers needed for that 
test instance. In this way a new JVM is created each time and the results are independent.
In fact, running all the tests in the same \texttt{for}-loop, or using \texttt{gradle run}
will use the same JVM and the latter instances will benefit from a fully-warmed up JVM.

Seven execution times are gathered and outliers, the biggest and the lowest times, are removed
to filter out eventual GC or OS slowdowns. An average is calcuated with the remaining five results.

Speedup is the ratio of improvement in speed execution between the sequential and parallel versions of a program,
it's calculated as follows:

\[
    S = \frac{T_s}{T_N}
\]
Where $N$ is the number of processors used.

We can observe that the lowest execution time is at 16 assigned cores.
The same times are plotted in logarithmic scale in Figure~\ref{fig:test-plot}.
\begin{figure}[H]
    \input{./figures/verification/test-plot.tikz}
    \centering
    \caption{Executions times over number of worker for each test}\label{fig:test-plot}
\end{figure}

We can see that, the more workers are added, the execution time trends downward. Although this
trend slows down towards 12 workers. This can be explained with Amdahl's law that
mathematically expresses the upper limit on speedup while using $N$ processors:

\[
    S = \frac{1}{(1-P)+\frac{P}{N}} \,\,\text{ with }\,\, 0\le P\le1
\]
where $P$ is the parallelizable part of the program.

The slowdown in performance gain from 12 cores and on is likely due to the fact that the non-parallel
part of the program ($1-P$) is still too great relative to the paralellizable part, and adding 
more workers is only granting a marginal gain.

We can plot the speedup of the test instances against the ideal speedup, achieved if a program could be totally
paralellizable ($P=1$), thus giving a linear increase in performance.

\begin{figure}[H]
    \input{./figures/verification/amdahl-plot.tikz}
    \centering
    \caption{Speedup of Poool against the ideal speedup}\label{fig:amdahl-plot}
\end{figure}

As we anticipated, the speedup is far lower than the ideal one, this is due to the sequential part
of the program taking its toll (mainly the spatial grid's creation and many more small wastes).

Another figure that we can calculate and that further explains the situation is Efficiency,
that measures how effectively each processor is used.
It's calculated by normalizing the speedup value to the number of processors $N$:
\[
E = \frac{S}{N}
\]

\section{Conclusion}
\subsection{Perfomance improvement}
Performance was seriously improved from the sequential version, offering a pleasing gaming experience
of about 45 FPS on a 8-core machine. Much more improvements could be done to the architecture and
minor efficinecies could be implemented.
\subsection{Further improvements}
The overall structure of the controller loop is solid but a refactoring is certainly necessary
to improved the components behavior. For instance a better way of work assignment could be implemented
both to prevent resource sharing and to give an equal amount of work, better exploiting the resources.
