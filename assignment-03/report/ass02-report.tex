\part[Smart Home Alarm System]{Smart Home Alarm System \\ \LARGE{Asynchronous message passing based on actors using Apache Pekko}}

\section{Analysis}
It is required to build an application that simulates a Smart Home Alarm System by using actors. The main requirement is to use 
asynchronous message passing; to do so, the Scala Pekko Library is used, as it offers a framework that focuses on the actor's state
and its relative behavior and handles autonomously the delivery of messages.

An actor is an entity that encapsulates a state, behavior and a control flow. It is strongly related to OOP, we could say that actors
are concurrent versions of objects as they interact with each other and encapsulate a state. 
Each actor changes behavior, the way it reacts to messages, according to it's previous behavior and an internal state.

The aspects that we have to keep in mind while designing and implementing the actors are:
\begin{enumerate}
  \item message passing is asynchronous, no assumptions can be made the deliveries apart that they will eventually happen
  \item message delivery order can't be assumed
  \item actors are purely reactive
  \item only messages can be exchanged and other actors cannot access the state of other actors
\end{enumerate}

Some of the aspects mentioned are already handled by Pekko, others have to be addressed in the desing phase.

\section{Design}
Seen the base properties of actors, we can design each actor as a Finite State Machine (FSM) that will handle some messages relative to the 
current state while other messages will be unhandled or have no effect on the state.

First, each actor can create other actors as their children, with this in mind a \textit{tree} is created, 
with at the root the first actor spawned.
Seen the optional requirement of dividing the alarm system into \textbf{zone}s, we have thought to distribute the alarm's behavior
in different \textbf{Zone Actors} so that each zone has an independent behavior. All the Zone Actors report to the 
\textbf{Alarm Actor} that aggregates all the states and passes messages to the zones.
An actor to interact with the system, the \textbf{Keypad Actor} is introduced.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\textwidth]{figures/alarm/actor-system.png}\label{fig:actor-system}
  \caption{The actor system with three zones configured}
\end{figure}

Each \textbf{Zone Actor} is implemented as a FSM that has the following states:
\begin{enumerate}
  \item \textbf{Armed}: reacts to sensor triggering and enters Entry Delay
  \item \textbf{Entry Delay}: gives some time before alarming the system
  \item \textbf{Alarm}: waits to be disarmed
  \item \textbf{Exit Delay}: when the alarm is set, doesn't react to sensor triggering for some time
  \item \textbf{Disarmed}: does nothing, waits to be armed
\end{enumerate}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.80\textwidth]{figures/alarm/zone-fsm.png}\label{fig:zone-fsm}
  \caption{The Zone Actor's FSM}
\end{figure}


The \textbf{Alarm Actor} aggregates all the zone actors, sends commands to them and enters alarm state when one or more zone
actors are alarmed. In the analogy with a real alarm system, this is the central alarm and the speakers that produces sound.
It only has two states: either it stays active and passes throught messages from the keypad to the zones and viceversa
or is in alarm state if one or more armed zones are triggered.

The \textbf{Keypad Actor} is the interaction point for the user, it stores a pin and talks to the Alarm Actor, receiving updates
on the alarm state that have to be communicated to the user.

\section{Implementation}
In Apache Pekko each actor can be represented by its behavior. A behavior is a function that regulates the reactions to
some (or all) types of messages.
Each actor spawns its own childs and eventually passes its own reference (\texttt{ActorRef[Command]}) to them to 
enable communication.
For some behaviors, some message types remain unhandled and are ignored. 

An interesting mechanism is used in the \texttt{AlarmUser} object. This object is used to represent the user as an actor
so that it can communicate with the keypad through messages. 

To verify the application, sensors have to be triggered and seen that they are spawned directly by ZoneActors, their reference
is not directly available the Receptionist is used.
The Receptionist is one of the two ways of discovering actors references in Pekko and its used when is impossible to directly
access the actor's reference, as in this case.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\textwidth]{figures/alarm/sensor-registration.png}\label{fig:sensor-reg}
  \caption{The Zone Actor's FSM}
\end{figure}

First, the actor that we want the reference from has to register to the Receptionist instance,
as shown in Figure~\ref{fig:sensor-reg}, by creating a \texttt{ServiceKey} and passing it.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{figures/alarm/sensor-find.png}\label{fig:sensor-find}
  \caption{The Zone Actor's FSM}
\end{figure}

Then, the AlarmUser registers a message adapater for the Receptionist reply, so that when a message arrives from the
Receptionist, which type is not included in the possibile message, it can be \textit{converted} into a 
message readable by the alarm user.
Finally the Alarm user can send a \texttt{Find} message to the receptionist containing the same \texttt{ServiceKey}
and the Receptionist will reply with a Set of all the registered actors for that key, as shown in Figure~\ref{fig:sensor-find-2}.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.85\textwidth]{figures/alarm/sensor-find-2.png}\label{fig:sensor-find-2}
  \caption{The Zone Actor's FSM}
\end{figure}

In this case the Receptionst is not interrogated immediately, but a timer is set (Fig.~\ref{fig:sensor-find})
so that all the actors can be created
and we are sure that by the time the \texttt{Find} message is sent all the actors have registered.

\part[Odd and Even]{Odd and Even \\\LARGE{Synchronous message passing in Go}}

\section{Analysis}
It is required to build an application where actors interact to play a tournament of \textit{odd and even}.

The required programming language is Go, it serves perfectly the purpose due to its native support for easy management of concurrency.

The requirement is to use synchronous message passing so actors are required to actively wait for a message to be delivered
and no memory has to be shared between the components.

\section{Design}
The idea is one of a centralized control, where a few components are considered the source of truth for the 
game state, therefore consensus mechanisms are not needed to determine which player won a match or the entire tournament.

The design expects the following components:
\begin{enumerate}
  \item a \textbf{Chief} that controls the tournament, matches the players against each other and keeps track of the
    remaining players after each round.
  \item some \textbf{Refree}s, tasked by the Chief to direct a match, coordinating the players, certifying and communicating
    the result to the Chief
  \item some \textbf{Player}s, waiting for a match to start and eventually terminating when loosing.
    Players are passive components reacting to the messages arrived.
\end{enumerate}

For the communication, messaging channels have to be created together with a messaging protocol so that the components know
what to expect from each other. 

The messages that will be exchanged will be:
\begin{enumerate}
  \item a Refree requesting a choice between odd and even to a Player
  \item a Refree requesting a number to a Player
  \item a Player returning a number to a Refree
  \item a Player returning its choice to a Refree
  \item the Chief tasking a Refree to direct a match
  \item a Refree reporting the winner and loser of a match to the Chief
  \item the Chief asking a player to terminate after loosing.
\end{enumerate}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.87\textwidth]{figures/channels.png}\label{fig:chan}
  \caption{Summary of channels and messages used between the components}
\end{figure}


The Chief, having to manage in parallel but in a synchronous fashion different matches at the same time, is modeled as a
Finite State Machine, as shown in Figure~\ref{fig:fsm}. The states represents the phases of a round, during which:
\begin{enumerate}
  \item matches are created by coupling together players, Refrees are tasked to direct said matches
  \item the Chief synchronously waits for the Refrees to report the results, updating the tournament's state
  \item the round passes or the tournament eventually ends, declaring a winner 
\end{enumerate}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.80\textwidth]{figures/chief-fsm.png}\label{fig:fsm}
  \caption{The Chief's Finite State Machine}
\end{figure}


\section{Implementation}
As said, Go is a perfect language to develop this kind of design.

Firstly, typed \texttt{struct}s are written to capture the necessary data for each component. It has been decided to give channels 
to the Chief and the Players, as Refree will be short lived and can use the other component's channels to communicate. 

\begin{figure}[H]
  \centering
  \includegraphics[width=0.90\textwidth]{figures/player.png}\label{fig:player}
  \caption{The player reacts to messages sent on his channel}
\end{figure}


Regarding channels, its important to note that Go offers buffered and unbuffered channels, the latter are intrinsically synchronous as there's
no buffer to store the unread messages. By creating this kind of channels, goroutines will have to block upon sending a message until some other
goroutine on the other side reads it. This satisfies the requirement for synchronous communication.

To read from multiple channels the standard \textit{for-select idiom} is used. In fact, if a \texttt{select} statement
is used to check multiple conditions, the goroutine will stop until one of them is true and will execute the relative code.
In this case the goroutine will stop until one or more messages are available on the channels checked.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.90\textwidth]{figures/for-select.png}\label{fig:fs}
  \caption{The \texttt{for-select} idiom used to check two channels}
\end{figure}

In the case of Figure~\ref{fig:fs}, the goroutine loops indefinitely and checks both \texttt{p.ctx.Done()} and \texttt{p.ch} channels.
In this way it can either capture a closure from the first channel to terminate (generated by the context) or a message coming from
the other components.
